Goal 持久化 · 最终技术方案

修一个「goal 已结束却在唤醒时反复冒提示」的 bug，前三轮越修越复杂（CAS / 跨进程锁 / watermark / 双 sidecar），每轮复核都生新 blocker。本方案止损回到简洁形态，参考 Codex CLI / Claude Code / OpenCode 三家真实做法定案。基线 c3e9020b。

核心结论

一个 state.json.goalLifecycle 状态字段搞定，不造任何并发协议

每个 session 只有一个 goal 权威来源：带 version 和不可变 goalId 的判别联合 active | waiting | terminal。所有变更走「读最新 state → 只改本领域字段 → tmp+rename」。删掉 双字段 activeGoal+goalTerminal、objective+setAtMs 脆弱 identity、whole-state tombstone merge。不引入 revision / CAS / watermark / 锁 / sidecar / journal。

要不要回退那套复杂机制 已确认没进 main
main 是干净的单 tombstone 版，复杂协议只活在被否决的实验 diff 里
并发契约
同 session 多进程并发写 = 明确不支持（本地单用户可接受）
复杂度
三处持久化真相 + 自研文件协议 → 一次 state.json 原子状态转换

2当前 main 现状（起点）

好消息 复杂机制没进 main

对 packages/core/src 精确搜索，当前 HEAD 没有：

  • ❌ goalRevision / revision allocator
  • ❌ CAS / expected revision
  • ❌ goalTerminalWatermark
  • ❌ goal lockfile / lease
  • ❌ goal-lifecycle.json / goal-terminals/ journal
  • ❌ legacy hash bridge / terminal 历史数组

→ 重实现不是「拆一套已合入的复杂协议」，而是「替换干净但表达力不足的单 tombstone + 收紧写入边界」。

待清债务 干净但不够好

  • 双字段 activeGoal? + goalTerminal? 分开存 types.ts:229-305
  • identity 靠 objective + setAtMs（脆弱）goal.ts:52-76

  • title 是 fire-and-forget，延迟回调改整份 session.state 再保存 → 进程内 stale writeback 真实来源 engine.ts:2263-2294

forced termination 清 goal、waiting 不清、arm guard —— 这些行为已正确，是必保留绿线。

3推荐数据模型

// 唯一权威字段，替换 activeGoal + goalTerminal
type GoalTerminalReason =
  | "judge_met" | "complete_goal" | "cancel_goal" | "user_cleared"
  | "stop_blocks_exhausted" | "token_budget_exhausted"
  | "time_budget_exhausted" | "max_turns_exhausted";

type GoalLifecycleV1 =
  | { version:1; goalId:string; phase:"active";   config:GoalConfig; updatedAtMs:number }
  | { version:1; goalId:string; phase:"waiting";  config:GoalConfig; updatedAtMs:number;
      waitingSinceMs:number; waitingFor:"finite_background_work" }
  | { version:1; goalId:string; phase:"terminal"; config:GoalConfig; updatedAtMs:number;
      terminal:{ reason:GoalTerminalReason; atMs:number } };

interface SessionState {
  goalLifecycle?: GoalLifecycleV1;          // 新写只输出这个
  activeGoal?: GoalConfig;                  // 仅 legacy 迁移期 decode，新 writer 不再输出
  goalTerminal?: GoalTerminal;              // 同上
}

关键不变量

  • 唯一权威：goal 状态只看 goalLifecycle，禁止从其他字段/session status 拼。
  • ID 即身份：每次显式 set/restart 都新建 goalId（含 same-objective 改预算）；A/B 是否同一只看 ID。
  • 可 arm 集：active 与 waiting 可进下一 run；terminal 永不自动 arm。
  • 正交：status=completed / manual stop 都不能推导 terminal。
  • fail-closed：未知 version / 非法 phase → 不 arm、不覆写原文件、报可诊断错误。

4生命周期状态机

none / terminal—— explicit set (新 goalId) ——▶active
active—— 有限后台任务让出 ——▶waiting
waiting—— 后台唤醒 / bare send ——▶active (同 goalId)
active/waiting—— judge met / complete / confirmed cancel / user clear ——▶terminal
active—— forced stop (4 种) ——▶terminal
active—— manual Stop / aborted_streaming ——▶active (不转移)
terminal—— bare send / resume / 通知唤醒 ——▶terminal (不 arm)

最后一行就是原 bug 的正解：terminal goal 被唤醒时保持 terminal、不重新 arm，round 不再从 1 重来。

5SessionManager 领域更新 API

私有同步原语 updateState(sid, mutator)：读最新 → 严格 decode/迁移 → 只改本领域字段 → 同目录写 tmp → rename。同进程 read/rename 之间不跨 await，天然串行。不是跨进程 CAS，不接受 expected revision。

API可改字段条件 / 返回
setGoalgoalLifecycle总是生成新 goalId，返回 committed active record
markGoalWaiting(expectedGoalId)goalLifecycleID 匹配且 active 才转 waiting，否则 no-op
armGoal(expectedGoalId)goalLifecyclewaiting→active；active 幂等；terminal/no-match 不 arm
terminateGoal(expectedGoalId, reason)goalLifecycleID 匹配且 armable 才转 terminal；幂等
clearCurrentGoalgoalLifecyclearmable→terminal(user_cleared)；terminal/无 goal 返回 false
readArmableGoal—active/waiting 返回 view；terminal/none/invalid 返回无
updateRunProgress / completeRunstatus/usage/turn/cost不携带 goal/title/workspace 旧值
setTitle / setWorkspace / resetUsage各自单领域fire-and-forget title 只能 setTitle，不覆盖 goal
消除 stale writeback：运行期业务代码禁止再把整份 SessionState 传给 public whole-state saveState。expectedGoalId 是进程内延迟回调的 identity guard —— 旧 run 的 A 回调若 ID 不匹配当前 B，transition 直接 no-op，改不动 B。
