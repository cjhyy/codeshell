# Core Review Findings

## P0 Critical

No P0 findings.

## P1 Important

- [ ] [P1] packages/core/src/plugins/installer/installFromSource.ts:37 — remote plugin `#subdir` is joined under the temp clone without rejecting `..` segments, while `parseSource.ts:63` accepts the fragment verbatim, so a malicious git source can escape the clone and install/copy an arbitrary local directory — validate subdir as a relative in-repo path and realpath-check containment under the clone root before `installPluginFromPath`.
- [ ] [P1] packages/core/src/plugins/pluginInstaller.ts:118 — marketplace string sources may be absolute or `../../` paths and are copied to the plugin cache with no containment check against the marketplace checkout — reject absolute/out-of-tree source paths or realpath both `marketplaceInstallLocation` and `src` and require `src` to be a strict child.
- [ ] [P1] packages/core/src/plugins/pluginInstaller.ts:157 — marketplace `git-subdir` sources join the manifest-controlled `path` under the cloned repo but do not reject `..`, allowing the copied source to escape the repo tempdir — validate `source.path` as a safe relative path and realpath-check containment under `tmp/repo` before `cp`.
- [ ] [P1] packages/core/src/tool-system/executor.ts:595 — the path policy approves relative args after resolving them against `ctx.cwd`, but the executor still passes the original args to handlers that use raw paths such as `read.ts:41`, `write.ts:36`, `edit.ts:45`, `notebook-edit.ts:64`, and `lsp.ts:61`, so approval can cover one file while the tool reads or writes a different `process.cwd()`-relative file — normalize declared path args before execution or make every path-taking builtin resolve through `ctx.cwd`, with tests where `process.cwd() !== ctx.cwd`.
- [ ] [P1] packages/core/src/tool-system/builtin/generate-video.ts:328 — `GenerateVideo` path policy checks `image`/`images` against `ctx.cwd` (`builtin/index.ts:169`), but upload normalization passes raw strings to `image-uploader.ts:41`, which reads from `process.cwd()`, so a local image can be approved as one file and uploaded from another path — resolve local image args to absolute `ctx.cwd` paths before calling the uploader.
- [ ] [P1] packages/core/src/run/FileRunStore.ts:45 — `runId` is joined directly into the runs directory and reused by destructive/read paths (`delete` at `FileRunStore.ts:174`, lock target at `RunLock.ts:161`, heartbeat path at `Heartbeat.ts:138`), unlike sessions which have `assertSafeSessionId`, so any public/store caller with a path-shaped run id can traverse outside the runs root — add and apply an `assertSafeRunId` single-segment guard in store, lock, heartbeat, and RunManager entry points.
- [ ] [P1] packages/core/src/settings/manager.ts:325 — dotted setting writers walk inherited objects, so `config_set("__proto__.polluted", value)` reaches `Object.prototype` through `saveUserSetting` and protocol only blocks protected roots at `server.ts:1456` — reject empty, `__proto__`, `prototype`, and `constructor` segments in all dotted-key read/write/delete paths and only descend into own plain objects.
- [ ] [P1] packages/core/src/run/RunApprovalBackend.ts:76 — the run approval backend stores only one pending approval; concurrency-safe ask-gated tools can start in parallel (`streaming-tool-queue.ts:37`, read-only MCP tools are `permissionDefault: "ask"` and `isConcurrencySafe` at `mcp-manager.ts:254`), so a second approval overwrites the first promise and leaves the first tool/run hanging forever — queue/serialize approvals like `InteractiveApprovalBackend`, or reject/supersede the older approval and clear its timeout.
- [ ] [P1] packages/core/src/run/RunManager.ts:204 — `resume()` resolves an in-process waiting approval even when `store.getApproval(runId, approvalId)` returns null or a non-pending record, so any stale/wrong approval id can still approve or reject the suspended tool; the id is also filename-joined at `FileRunStore.ts:228` without segment validation — require a safe approval id, require it to match `run.latestApprovalId`, require the loaded record to belong to the same run and be pending, and reject mismatches before `handle.resolveApproval`.
- [ ] [P1] packages/core/src/run/RunManager.ts:270 — the comment on `resolvingRuns` says resume and cancel are serialized (`RunManager.ts:79`), but `cancel()` never checks or sets that guard, so cancel racing resume can both pass the waiting-state check and drive duplicate/conflicting transitions — wrap cancel in the same per-run resolving guard or share a `withResolvingRun` helper.

## P2 Minor

- [ ] [P2] packages/core/src/run/RunManager.ts:224 — waiting-input resume checks `input?.userInput`, so an intentionally empty string is treated as missing input and the live suspended ask throws the mismatch path — check for an own `userInput` property with `typeof input.userInput === "string"` instead of truthiness.
- [ ] [P2] packages/core/src/run/RunApprovalBackend.ts:77 — approval timeout timers are never cleared when `resolveApproval()` succeeds at `RunApprovalBackend.ts:92`, leaving one unref'ed timer/closure alive until the timeout window for every approved request — store the timer in the pending record and `clearTimeout` it on resolve, supersede, and timeout.
- [ ] [P2] packages/core/src/session/session-manager.ts:108 — creating a session with an explicit id uses recursive mkdir and overwrites `state.json`, but `Transcript` preserves an existing `transcript.jsonl` at `transcript.ts:22`, so a reused/raced explicit id mixes a new session state with old transcript events — fail if the session dir already exists, or use exclusive creation and truncate transcript only when overwrite is explicitly intended.
- [ ] [P2] packages/core/src/session/session-manager.ts:490 — session preview parsing treats every `role: "user"` message as user input and ignores the transcript `injected` flag, despite `transcript.ts:43` documenting injected user-role reminders as synthetic — skip `event.data.injected === true` when deriving the latest user preview.
- [ ] [P2] packages/core/src/run/FileRunStore.ts:58 — JSON writes use a fixed `filePath + ".tmp"` name, so concurrent writers in different workers updating the same run file can clobber or delete each other's temp file before `renameSync` — include pid plus a random/UUID suffix in temp names, matching the safer settings/session write pattern.
- [ ] [P2] packages/core/src/run/Heartbeat.ts:41 — calling `start(runId)` twice overwrites the timer map entry without clearing the old interval, so `stop(runId)` only clears the latest timer and leaks the first heartbeat writer — make `start` idempotent or call `stop(runId)` before creating a new interval.
- [ ] [P2] packages/core/src/tool-system/mcp-manager.ts:348 — `desiredByOwner` deliberately keeps closed sessions' desired MCP server sets forever, so removed/disabled project MCP servers can stay connected until worker exit — add an owner teardown/unregister hook on Engine/session close and reconcile after removing that owner's desired set.
