凭证服务 Partition 修复前后对照

Bug 的核心：浏览器面板把 cookie 写入每个 chat session 的专属 persist:browser:<bucket>，但凭证服务此前读写的是另一个共享 persist:browser，所以抓取和恢复都落到了错误分区。

当前逻辑

修复前：凭证服务不看当前面板，直接读写 persist:browser。

对照

修复前 面板写 persist:browser:<bucket>，凭证服务读 persist:browser，两个集合不相交。
修复后 凭证服务跟随当前 active guest，直接用它的 Electron Session 对象读写 cookie。

小结

  • 根因：per-session 浏览器面板使用 persist:browser:<bucket>， 旧凭证服务却硬编码 persist:browser，导致抓取预览为空、账号导回不到真实登录分区。
  • 修复思路：凭证服务跟随当前活跃浏览器面板的 guest，拿该 guest 对应的 Electron Session 对象读写 cookie；没有 active guest 时才回落到共享 persist:browser。
  • 关键约束：不要依赖反推 partition 字符串；Electron 的 Session 对象本身拿不到可靠的 partition 字符串，直接传递/使用对象更稳。
  • 恢复策略：restore 默认从 clear 改为 merge，避免导入某个站点 cookie 时误清掉目标分区里其他站点的登录态。
本页只模拟凭证服务的读写目标选择和 cookie 数量变化，不接真实 Electron、浏览器内核或加密存储。
