## B. 新一轮深挖发现(进行中)

### B-1 🔴 path-policy 敏感文件正则过宽 → 误杀常见源码文件的写入
`tool-system/path-policy.ts:91-101` `SENSITIVE_FILE_PATTERNS` 含 `/auth/i`、`/token/i`、`/credential/i`、`/secret/i`,在 `matchSensitiveFile`(:289)用 `re.test(basename)` **子串匹配**。任何文件名含这些子串即判为敏感 → 写入 `decision:"deny"`(:367,workspace 内外都拒,且 deny 在审批前短路)。

**实测命中(node 验证)**:`token-counter.ts`(本仓自己就有!)、`authController.ts`、`useAuth.tsx`、`auth.ts`、`refreshToken.ts`、`credentials.ts`、`mySecret.json`、`tokenizer.ts`、`oauth-handler.ts` 全部 **写入被拒**。

**影响**:非 bypass 模式(default/acceptEdits/plan)下,Edit/Write/ApplyPatch(edit.ts:57 / write.ts:38 / apply-patch:95 都调 `enforcePathPolicyWithApproval(...,"write")`)**无法编辑任何文件名含 auth/token/credential/secret 的源码文件**——编码 agent 的核心场景被打断。bypassPermissions(:445)才跳过。

**非有意**:path-policy 测试只覆盖 `~/.ssh/config` 真敏感目录,没有任何用例锁定"源码文件子串"行为 → 是过度匹配而非契约。

**建议修法**:把这些宽松子串收紧为「真凭据文件」约定——例如仅匹配整名 `^(\.?secrets?|credentials?|\.?auth)\.(json|ya?ml|txt|ini|conf|toml)$` 之类,或加 word-boundary + 已知凭据文件名清单;保留 `.env*`/`id_rsa`/`*.pem`/`*.p12`/`*.pfx` 这些确定无误的。务必加回归测试:`auth.ts`/`token-counter.ts` 写入应 allow。 🔴

### B-2 🟢 turn-loop 工具摘要 fire-and-forget 缺外层 catch
`engine/turn-loop.ts:813` `import("./tool-summary.js").then(...)` 无 `.catch`。`generateToolUseSummary` 内部已 try/catch 返回 null,但**内层 `.then(summary => onStream(...))` 里若宿主 `onStream` 抛错** → unhandled rejection(Node 可能噪声/崩溃)。低危。建议补 `.catch(()=>{})`。 🟢

### B-3 🟡 (复核中)turn-loop maxToolCallsPerTurn 静默丢弃超额工具调用
`engine/turn-loop.ts:766` `response.toolCalls.slice(0, maxToolCallsPerTurn)`——模型一轮发超过上限的工具调用时,超额部分**静默丢弃**,不给模型任何 `<system-reminder>` 说明"只执行了前 N 个"。assistant/tool_result 配对一致(无 API 错误),但模型会误以为全部执行了。建议:截断时注入提醒。**默认值 = 10**(engine.ts:1775)。现代模型常一轮发 >10 个并行工具调用(如一次读 15 个文件),触发概率高 → 5 个被静默丢弃,模型以为全跑了。**确认为真问题**。建议:提高默认值(如 20+)或截断时注入 `<system-reminder>` 告知"仅执行前 N 个,请重发其余"。 🟡

### B-4 🟢 OpenAI 流式工具调用 id 仅取首个 delta
`llm/providers/openai.ts:500-511`:`toolCallsMap` 按 `index` keying,首次见到某 index 时 `id: tc.id ?? ""`,后续 delta 只 append `args`、**从不更新 id**。严格 OpenAI 总在首 delta 带 id 没问题,但部分 OpenAI 兼容端点可能在later delta 才给 id → id 留空 `""` → 下游 tool_use/tool_result 配对失败(orphan)。低危(仅非标准端点)。建议:每个 delta 若带 id 就回填。 🟢

### B-5 🟡(决策项)settings "managed" 语义 = 最低优先级,与企业策略惯例相反
`settings/manager.ts:4,71-72`:`settings.managed.json` 被当作**最低优先级**(priority 0,被 user/project/local/flag 全部覆盖),注释明确写"Managed (lowest priority)"。但 `managed` 这个命名在 Claude Code / 企业策略惯例里是**管理员锁定的最高优先级**(IT 强制、用户不可覆盖)。当前实现下,若 org 想用 managed 强制 `permissionMode` 不许 bypass、强制 sandbox,用户改自己的 `settings.json` 就能轻松绕过。**是有意设计(注释明确),但语义与命名/惯例冲突**——需你确认:managed 到底是"出厂默认"(现状)还是"企业策略"(应最高优先级)。若是后者 = 🔴 安全策略可绕过。 🟡决策

### B-6 🟢 transcript.toMessages 乱序 transcript 下 tool_use 可能后接 user 文本
`session/transcript.ts:100-112`:tool_result 仅在 `lastMsg.role==="user" && Array.isArray(content)` 时追加,否则新建 user 消息。正常流程 tool_result 紧跟 assistant 没问题,且 `repairToolResultPairs`(:212 load 时调)+ `patchOrphanedToolUses`(engine.ts:1252)已兜底。仅在异常乱序 transcript(result 落在 user 文本之后)才可能产生 tool_use→user文本→tool_result 的非法序列。防御已较完备,低危。 🟢

## C. 已核查为「健壮/非 bug」的关键路径(给你信心)

- **turn-loop 中止处理**:loop-top + manageAsync 后 + 模型调用 catch + 续写循环 + 外层 try/catch 多处查 `signal.aborted`,abort 一律以 `aborted_streaming` 正常返回不抛错。健壮。
- **ChatSession.cancel / pump**:§5.1 担心的"engine 吞 abort 致 cancel 静默成功"——实测 `Engine.run` abort 时**正常返回** `reason:"aborted_streaming"`(非抛错),caller 可凭 reason 区分;`cancelledActive` 的 catch 吞错路径基本是死路但无害。非 bug(注释稍漂移)。
- **StreamingToolQueue.drain**:结果按 callOrder 返回,tool_use/tool_result 顺序对应,无错配。
- **patchOrphanedToolUses**:answeredIds 预扫 + 幂等 + O(n),resume 修补正确。
- **MCP 输出**:`wrapMcpOutput` 两处调用点都包裹 untrusted 标记,注入防御到位;`wrapUntrustedInput` 中和闭合标签。
- **Bash 分类**:compound(`&&`)、重定向、pipe-to-shell 都走 `classifyBashCommand`/`scanShellCommand`,`mkdir /tmp && rm -rf /` 不会漏判。
- **会话权限缓存**:按操作而非工具名 keying(批准 git 不会放行 rm)。
