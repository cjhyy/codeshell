# Fix Batch 10 - 2026-07-08

## FIX 1 - CDP `pressKey()` printable text payload

Evidence ref: `docs/archive/todo/arch-plans-and-verification-2026-07-08.md`, section
`NEEDS-VERIFICATION resolutions` / `Printable CDP pressKey() text payload`.
The verified bug was that printable `pressKey()` events reached
`Input.dispatchKeyEvent` without `text` or `unmodifiedText`, so printable keys
such as `a`, `A`, `1`, `-`, and `Space` did not insert text, while shortcuts
such as `Control+a` still worked.

Commit: `c72df028` (`fix(cdp): send text for printable key presses`)

Test added: `packages/cdp/src/driver.test.ts`,
`CdpActionsDriver.pressKey > sends text payloads for printable keys only`.

Verification: `bun test packages/cdp/src/driver.test.ts` passed before commit
with `32 pass`, `0 fail`.

## FIX 2 - CDP package npm publish consistency

Evidence ref: `docs/archive/todo/arch-plans-and-verification-2026-07-08.md`, section
`NEEDS-VERIFICATION resolutions` / `@cjhyy/code-shell-cdp publishability`.
The verified mismatch was that CDP was public and included in release
version checks, but the npm publish workflow omitted it and npm returned E404
for `@cjhyy/code-shell-cdp` while core, TUI, and meta existed.

Decision: root/core/TUI configs do not use CDP as a runtime dependency, and
desktop lists CDP as a dev dependency for bundled build-time use. The consistent
fix is to mark `packages/cdp` as private/non-npm instead of adding it to the
npm publish order.

Commit: `df8e9e83` (`fix(infra): keep cdp out of npm publish set`)

Test added: `packages/core/src/release-workflow.test.ts`,
`release workflow guards > version-checked npm packages are published unless explicitly private`.

Verification: `bun test packages/core/src/release-workflow.test.ts` first failed
with `@cjhyy/code-shell-cdp is version-checked and public but not published`,
then passed before commit with `3 pass`, `0 fail`.
