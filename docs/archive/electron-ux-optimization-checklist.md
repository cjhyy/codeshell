# Electron 体验优化清单

## 文档目的

这份文档是给后续 AI 接手 CodeShell Electron / 桌面端体验优化时看的。重点不是底层架构重构，而是记录当前体验上明显不顺手、不美观、需要统一设计和实现的地方。

后续 AI 应优先从用户可感知的体验问题入手：组件统一、样式改善、配置流程简化、输入能力增强。

## 总体优化方向

1. 统一 Electron 设置页/配置页里的基础交互组件，避免每个地方样式不一致。
2. 优化 MCP、连接、Skills 等配置区域的视觉层级和操作流程。
3. 减少用户手动填写配置的负担，尽可能支持自动识别、自动解析、批量管理。
4. 输入框需要支持图片等多模态输入，并针对非多模态模型设计降级策略。
5. 所有体验优化都应该有明确的验收标准，避免只做“看起来改了”的表面调整。

## P0：优先优化

### 1. Select / 下拉框组件统一优化

#### 当前问题

现在的下拉框太难看，视觉和交互体验都不理想。所有 `select` 都需要统一优化，不能继续使用零散的原生样式或各处手写样式。

#### 目标

做一个统一的下拉框组件，替换所有现有 select 场景。

#### 优化要求

- 统一视觉样式：圆角、边框、hover、focus、disabled 状态一致。
- 暗色模式下不能突兀。
- 下拉菜单层级、阴影、间距要更现代。
- 支持键盘导航。
- 支持搜索型 select，用于选项很多的场景。
- 支持空状态、加载状态、分组选项。
- 后续页面不要再直接使用原生 `<select>`。

#### 后续 AI 执行建议

- 先全局搜索现有 `select` 使用位置。
- 抽象一个基础 Select 组件。
- 逐步替换：设置页、模型选择、MCP 配置、Skill 配置、连接配置等。
- 不要只改 CSS；要让交互状态也统一。

#### 验收标准

- 所有页面不再出现突兀的原生 select。
- hover/focus/disabled/loading/empty 状态都有设计。
- 选项很多时不会难用。
- 视觉上和 Electron 整体 UI 风格一致。

### 2. MCP 服务器配置区域样式优化

#### 当前问题

MCP 服务器那一块样式比较丑，信息层级和操作入口不够清晰。

#### 目标

把 MCP 服务器管理区域做成更清晰、更像“连接管理”的体验。

#### 优化方向

MCP server 列表建议卡片化展示：

- 名称
- 状态：启用 / 禁用 / 连接中 / 错误
- 类型：本地命令 / HTTP / SSE / stdio 等
- 最近错误摘要
- 可用 tools 数量

每个 MCP server 应该有清晰操作：

- 启用 / 禁用
- 编辑
- 测试连接
- 查看 tools
- 删除

错误信息要更可读：

- 不要只显示原始报错。
- 应该有简短解释和“查看详情”。

样式上避免一堆表单堆在一起：

- 列表态和编辑态分离。
- 高级配置折叠。
- 危险操作放到明显但不打扰的位置。

#### 后续 AI 执行建议

- 先定位 MCP server 配置 UI 代码。
- 梳理当前 MCP server 数据结构。
- 先做信息展示层级，再优化编辑表单。
- 测试至少覆盖：无 MCP server、一个正常 server、一个报错 server、多个 server、tools 很多的 server。

#### 验收标准

- 用户一眼能看出每个 MCP server 当前状态。
- 添加、编辑、删除、测试连接路径清楚。
- 错误状态不再显得混乱或吓人。
- 整体样式和其他设置区一致。

---

### 3. 连接 / Link / 搜索工具配置区域优化

#### 当前问题

“链接那一块”需要优化。当前像是用“搜索的 key + 搜索工具”的方式配置，体验不够直观。

#### 当前理解

这里可能指某个连接配置区：用户需要配置搜索 key、选择搜索工具，或者把外部搜索能力接入 CodeShell。当前流程太像裸配置项，不像一个完整的“连接/工具集成”体验。

#### 目标

把搜索相关配置从“填 key + 选 tool”优化成更清晰的连接配置流程。

#### 优化方向

把“搜索能力”作为一个连接项展示：

- Provider / 工具名称
- 是否已配置 API key
- 当前是否可用
- 测试搜索按钮

配置 key 时：

- 显示 key 用途。
- 显示保存位置说明。
- 支持隐藏 / 显示。
- 支持清除。

搜索工具选择：

- 不要让用户理解内部 tool 名称。
- 用人能看懂的名称和说明。
- 例如：Web Search、本地文档搜索、MCP 搜索工具等。

如果没有配置 key：

- 显示明确引导，不要只报错。

如果配置了多个搜索工具：

- 支持默认工具选择。
- 支持禁用某个工具。
- 支持测试每个工具。

#### 后续 AI 执行建议

- 先找到当前“搜索 key / 搜索工具”配置入口。
- 把配置项重命名为用户视角的连接项。
- 避免暴露太多内部 tool 术语。
- 如果底层仍然是 key + tool，也应该在 UI 上包装成连接管理。

#### 验收标准

- 用户知道自己正在配置什么搜索能力。
- 用户知道 key 是否已经配置成功。
- 用户可以测试搜索是否可用。
- 配置失败时有可理解的错误信息。

## P1：重要体验优化

### 4. 已安装 Skills 管理体验优化

#### 当前问题

已安装 skill 现在是平铺或不够聚合。例如 `document-skills:canvas-design` 这种 skill 应该按照前缀 / 来源进行聚合。用户希望能点击进入分组内部勾选具体内容，方便整体取消或批量管理。

#### 目标

把已安装 Skills 做成“分组 + 详情 + 批量选择”的管理体验。

#### 优化方向

##### 4.1 按 namespace / plugin 聚合

例如：

```text
document-skills
  - algorithmic-art
  - brand-guidelines
  - canvas-design
  - docx
  - pdf
  - pptx

superpowers
  - brainstorming
  - systematic-debugging
  - test-driven-development
```

##### 4.2 分组级操作

每个分组支持：

- 展开 / 收起
- 全选 / 全不选
- 批量卸载
- 批量禁用
- 查看来源
- 查看安装路径

##### 4.3 单个 skill 操作

每个 skill 支持：

- 启用 / 禁用
- 卸载
- 查看说明
- 查看触发条件
- 查看来源
- 复制 skill 名称
- 打开本地文件位置，如果 Electron 支持

##### 4.4 选择交互

用户可以：

- 勾选整个分组。
- 进入分组后勾选部分 skill。
- 取消整个分组。
- 对选中 skill 执行批量操作。

#### 后续 AI 执行建议

- 先梳理当前已安装 skills 的数据结构。
- 用 `namespace:name` 拆分出 namespace。
- UI 上先实现只读分组展示，再加批量选择和操作。
- 注意批量卸载是危险操作，需要确认弹窗。
- 不要把已安装 skill 和可安装 skill 混在同一个无层级列表里。

#### 验收标准

- `document-skills:canvas-design` 能归入 `document-skills` 分组。
- 用户可以一键选择 / 取消整个分组。
- 用户可以进入分组只选择部分 skill。
- 批量操作前有明确确认。
- 分组很多时仍然好浏览。

### 5. Skill 安装流程优化：支持 GitHub 地址自动识别

#### 当前问题

安装 skill 现在需要手动填写，太麻烦。用户希望可以直接给一个 GitHub 地址，系统自动识别能不能安装。

#### 目标

把 Skill 安装从“手动填配置”优化成“粘贴地址 → 自动识别 → 预览 → 安装”。

#### 推荐流程

```text
用户粘贴 GitHub URL
        ↓
系统识别类型
        ↓
检测是否包含 skill/plugin 结构
        ↓
展示可安装内容
        ↓
用户选择要安装的 skill
        ↓
安装并展示结果
```

#### 支持的输入类型

后续 AI 可以考虑支持：

- GitHub 仓库地址
- GitHub 子目录地址
- GitHub raw 文件地址
- GitHub release 地址
- 本地路径，作为后续扩展
- 压缩包 URL，作为后续扩展

#### 自动识别内容

系统应尝试识别：

- 是否是 skill 仓库。
- 是否是 plugin 仓库。
- 是否包含多个 skills。
- skill 名称、namespace、description。
- 是否已经安装。
- 是否有版本信息。
- 是否有冲突。

#### 安装前预览

安装前应该展示：

- 将安装哪些 skill。
- 来源 URL。
- 安装位置。
- 是否覆盖已有 skill。
- 是否需要信任该来源。
- 是否有潜在风险。

#### 安全注意事项

- 不能静默安装未知远程代码。
- GitHub 地址解析和安装之间要有确认步骤。
- 对已有 skill 覆盖要额外确认。
- 如果 skill / plugin 里包含脚本或 hook，需要明确提示。
- 不要自动执行远程仓库里的任意脚本。

#### 后续 AI 执行建议

- 先实现 URL 解析和预览，不要一开始就自动安装。
- 支持最常见 GitHub repo URL 即可作为第一版。
- 安装失败要给出原因：不是有效 GitHub 地址、仓库不存在、没找到 skill 结构、已安装、权限 / 网络失败。
- 之后再扩展到子目录、release、zip。

#### 验收标准

- 用户粘贴 GitHub repo URL 后，系统能自动识别是否可安装。
- 可安装时展示预览。
- 不可安装时说明原因。
- 用户不需要手动填写复杂路径。
- 安装前必须确认。

## P2：输入能力增强

### 6. 输入框支持传图片

#### 当前问题

当前输入框主要是文本输入。用户希望可以像其他 AI 工具一样传图片。但不同模型对多模态支持不同，例如 Codex 如何处理图片、DeepSeek 这种非多模态模型应该怎么处理，需要设计清楚。

#### 目标

输入框支持图片附件，同时根据当前模型能力决定如何发送或降级处理。

#### 需要调研的问题

1. Codex CLI / Codex 类产品是怎么处理图片输入的？
2. 当前 CodeShell 的模型能力元数据里是否能识别模型是否支持 vision？
3. OpenRouter 或当前 provider 返回的模型列表里是否包含多模态能力字段？
4. DeepSeek 等非多模态模型收到图片时应该：禁止上传、上传后提示模型不支持、自动 OCR、自动生成图片描述再发送、还是允许作为文件附件但不传给模型？
5. 图片是否要落盘缓存？缓存生命周期是什么？
6. 图片会不会进入 transcript / session 存储？
7. 图片是否需要支持拖拽、粘贴、文件选择三种入口？

#### 推荐设计

##### 6.1 输入框交互

支持：

- 点击按钮选择图片。
- 拖拽图片到输入框。
- 从剪贴板粘贴图片。
- 图片以附件 chip / thumbnail 展示在输入框上方。
- 支持删除单张图片。
- 支持多张图片，但第一版可以限制数量。

##### 6.2 模型能力判断

发送前判断当前模型是否支持 vision：

- 如果支持 vision：图片随消息一起发送。
- 如果不支持 vision：不要静默丢弃图片，明确提示“当前模型不支持图片输入”。

不支持 vision 时，给用户可选操作：

1. 切换到支持图片的模型。
2. 对图片进行 OCR / 描述后发送文本。
3. 仅作为本地附件保留，不发送给模型。
4. 删除图片。

##### 6.3 非多模态模型降级策略

对于 DeepSeek 这类非多模态模型，推荐第一版策略：

```text
默认不直接发送图片。
如果用户上传图片并选择非多模态模型：
  - UI 显示模型不支持图片。
  - 提供“提取文字/OCR 后发送”或“切换模型”。
  - 不允许无提示地把图片丢掉。
```

如果后续有 OCR / 图片描述能力：

- OCR 适合截图、文档、错误信息。
- 图片描述适合 UI 截图、视觉 QA。
- 生成的文本应该明确标注来源，例如：

```text
[Image OCR result]
...

[Image description]
...
```

避免模型误以为用户直接输入了这些内容。

##### 6.4 存储和隐私

需要明确：

- 图片是否保存到 session。
- 图片缓存目录。
- 什么时候清理。
- 是否会被上传到 provider。
- 用户是否能删除历史图片。
- 日志里不能写入 base64 图片内容。

#### 后续 AI 执行建议

- 先调研当前消息协议是否支持 image content block。
- 再看 provider adapter 是否支持多模态。
- 先做 UI 附件能力和模型能力判断。
- 最后再接 provider 发送。
- 不要一开始就把图片转 base64 塞进普通文本。
- 对非多模态模型必须有明确降级提示。

#### 验收标准

- 输入框可以粘贴 / 拖拽 / 选择图片。
- 图片上传后有可见缩略图。
- 可以删除图片。
- 支持 vision 的模型能收到图片。
- 不支持 vision 的模型不会静默失败。
- DeepSeek 等非多模态模型有清晰提示和可选降级路径。
- 图片不会污染日志或无控制地长期保存。

## 给后续 AI 的执行顺序建议

推荐顺序：

1. 先做统一 Select 组件。
2. 再优化 MCP server 配置区域。
3. 再优化连接 / 搜索工具配置区域。
4. 再做已安装 Skills 的分组和批量管理。
5. 再做 GitHub URL 安装 skill 的自动识别。
6. 最后做图片输入，因为它涉及 UI、模型能力、provider 协议、存储和隐私，复杂度最高。

## 注意事项

- 这是体验优化清单，不要一上来做大规模架构重构。
- 每个优化点都应该先找现有 UI 入口和数据结构。
- 不要只改样式，要补齐状态、空态、错误态、加载态。
- 涉及卸载、覆盖、远程安装的操作必须有确认。
- 涉及图片上传的能力必须考虑模型兼容性、隐私、缓存和日志。
