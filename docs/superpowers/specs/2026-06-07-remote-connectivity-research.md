# 远程控制连接架构调研 — IP 变动 / 长期连接 / 是否必须公网

> 2026-06-07 · 调研文档(不含实现)。回答:IP 变了怎么办?必须过一个公网固定地址吗?

## 现状

`remote-host-manager.ts`:桌面进程起一个 HTTP+WS server,`server.listen(port, bindHost)`,
`bindHost = resolveLanHost()`(本机局域网 IP,如 `192.168.1.23`,**故意不绑 0.0.0.0/127**)。
二维码编码 `http://192.168.1.23:<port>/mobile?pairing=<token>`,手机扫码连。

**两个真问题:**
1. **跨网不通**:手机必须和桌面在同一个 LAN。手机用蜂窝(4G/5G)、或在公司/咖啡馆网络 → 连不上。
2. **IP 漂移**:路由器 DHCP 重新分配(重启/租约到期/换网)→ 局域网 IP 变,旧二维码和手机里存的地址失效,要重新扫码。

**直接回答"必须过公网固定吗":不必须。** 公网固定 IP 只是其中一种(且对个人开发者最贵最不安全)。下面 5 种方案从"只解决 IP 漂移"到"任意网络可达"递进。

---

## 方案对比

### A. mDNS / Bonjour(`codeshell.local`)—— 只解决同网 IP 漂移

- **原理**:桌面用 mDNS 广播一个主机名(`codeshell-<id>.local`),手机连 `http://codeshell-xxx.local:<port>` 而非 IP。IP 漂移时名字不变,系统自动解析到新 IP。
- **可达范围**:仅同一 LAN(和现状一样)。**不解决跨网**。
- **成本/依赖**:零外部服务。桌面加一个 mDNS 库(如 `bonjour-service`)。iOS/macOS 原生支持 `.local`;Android 部分浏览器对 `.local` 解析不稳(需测)。
- **安全**:不变(仍是 LAN 内,配对 token + 设备密钥那套照旧)。
- **取舍**:**最小改动、最低风险**,直接消灭"IP 变了要重扫码"。但手机一旦离开家里的 Wi-Fi 就没用。

### B. 局域网内"重连即重新发现" —— 纯客户端兜底,不引依赖

- **原理**:手机存"上次的 host:port",连不上时不立刻报错,而是尝试一组候选(上次 IP、网关同段扫描、或回退到引导页让用户重扫)。配合一个固定的"发现端点"(桌面在多个网卡都 listen)。
- **可达范围**:同 LAN。比现状强一点(IP 在同网段内漂移能自愈),但跨网仍不行。
- **成本**:零依赖,纯前端逻辑 + 桌面 bind 调整。
- **取舍**:廉价兜底,但网段扫描慢且不可靠(企业网常禁)。**不如 A 干净**,可作为 A 的补充。

### C. 内网穿透 / Overlay VPN(Tailscale、ZeroTier)—— 跨网,推荐给个人开发者

- **原理**:桌面和手机都装同一个 overlay 网络客户端(Tailscale/ZeroTier),组成一个虚拟 LAN。每台设备拿一个**稳定的虚拟 IP**(Tailscale 的 `100.x.y.z` 或 MagicDNS 名 `mac.tailnet.ts.net`),无论物理网络怎么变都不变。手机连那个虚拟 IP/名即可,4G 也通。
- **可达范围**:**任意网络**(NAT 穿透由 overlay 处理,多数情况 P2P 直连,失败回退中继)。
- **成本/依赖**:用户两端各装一个 app + 登录同账号。CodeShell 不自建服务器,**借力成熟服务**。Tailscale 个人免费档够用。
- **安全**:**最好**——overlay 自带端到端 WireGuard 加密 + 设备授权,CodeShell 的流量天然只在自己的 tailnet 内,不暴露公网。
- **取舍**:依赖第三方 app + 用户得懂一点。但对"开发者自己用"这个受众,这是**性价比和安全双优**的答案。CodeShell 这边几乎零改动(只需检测/提示用户用 tailnet 名,二维码编码虚拟地址)。

### D. 自建/中继隧道(frp、Cloudflare Tunnel、ngrok)—— 跨网,但要一个公网入口

- **原理**:桌面跑一个隧道客户端,把本地 server 暴露成一个公网 URL(frps 服务器 / Cloudflare / ngrok 域名)。手机连那个公网 URL,任意网络可达。
- **可达范围**:任意网络。
- **成本/依赖**:**需要一个公网中继**——要么自己租 VPS 跑 frps(回到"要公网固定地址"),要么用 Cloudflare Tunnel(免费但要 CF 账号 + 域名)/ngrok(免费档有限速和随机域名)。
- **安全**:**风险最高**——你的 agent 控制面被映射到公网 URL,即使有配对 token,也等于把"能远程跑命令的入口"挂到了公网,被扫描/爆破的攻击面大。必须叠加强认证 + 速率限制 + 最好 mTLS。
- **取舍**:能力最强但安全负担最重、依赖公网入口。**不推荐作为默认**,只在"必须从完全不可控的网络访问且不能装 overlay"时考虑。

### E. 公网固定 IP / 端口转发 —— 不推荐

- **原理**:家里宽带配公网 IP + 路由器端口转发到桌面。
- **取舍**:多数家宽是 NAT 后的动态 IP / CGNAT,根本拿不到公网 IP;就算有,把 agent 控制面直接挂公网是**最危险**的做法。**明确不做。**

---

## 推荐路线(分层,按投入递增)

1. **先做 A(mDNS `.local`)** — 小改动直接消灭"IP 变了重扫码"这个最高频痛点,覆盖"在家同网"的主场景。风险低、零外部依赖。
2. **跨网场景官方推荐 C(Tailscale/ZeroTier)** — 不自己造轮子,文档里教用户两端装 overlay,CodeShell 只需:① 配对二维码能编码用户填的 tailnet 名/虚拟 IP;② 检测到非 LAN 地址时给安全提示。**这是"4G 也能连家里"的正解,且最安全。**
3. **D/E 默认不做** — 把 agent 控制面挂公网的攻击面不值得;真要做必须配 mTLS + 速率限制 + 审计,作为高级开关另谈。

**一句话结论给用户**:不需要公网固定 IP。同网下用 mDNS 名解决 IP 漂移;要跨网(4G 连家里)就两端装 Tailscale,既稳又安全,CodeShell 不碰你的密钥也不开公网口。

## 落地时的安全红线(任何方案都适用,呼应 P5 设计)

- 配对 token 一次性 + 短期有效;设备密钥存两端,认证按设备。
- 不自动外发密钥;不绕过桌面端自身的权限/审批(手机只是另一个前端,审批仍走同一套)。
- 跨网方案默认要求传输层加密(overlay 自带 / 隧道叠 TLS),不裸 ws 跨公网。
